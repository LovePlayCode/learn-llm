## 1. AI工程的兴起：演进链路

```
语言模型 → 大语言模型(LLM) → 基础模型(Foundation Model) → AI工程
```


| 阶段         | 关键变化                  | 驱动因素                           |
| ---------- | --------------------- | ------------------------------ |
| 语言模型       | 对语言的统计信息编码，预测下一个词     | 统计学方法                          |
| 大语言模型(LLM) | 模型规模扩大，能力涌现           | **自监督学习**：不需要人工标注，直接从互联网海量文本学习 |
| 基础模型       | 从纯文本扩展到多模态（图像、视频、音频等） | 多模态数据融合                        |
| AI工程       | 基于现成模型构建应用            | 模型即服务(MaaS)降低门槛                |


### 自监督学习——LLM规模化的根本原因

- **监督学习**：需要人工标注数据，成本高、耗时长（标注100万张图片需5万美元）
- **自监督学习**：模型从输入数据中自行推断标签，无需人工标注
- **关键逻辑链**：不需要人工标注 → 可以用互联网上海量文本 → 训练数据几乎无限 → 模型可以做得越来越大 → LLM诞生

> 句子 "I love street food." 可以自动产生6个训练样本，每个样本的输入是前序token，输出是下一个token。

## 2. 核心概念详解

### Token——文本与数字之间的桥梁

```
文字 → Token → Token ID（数字） → 向量 → 模型能处理
```

- Token是文本被切分后的基本单位，可以是一个词、子词或字符
- **为什么用Token而不是单词或字符？**
  1. Token允许模型将单词拆分为有意义的组成部分（如 cooking → cook + ing）
  2. Token数比单词少，减小词表，提高效率
  3. Token能帮助处理未知单词（如 chatgpting → chatgpt + ing）

### 自回归——大模型生成文本的核心原理

- **定义**：仅利用前序token来预测下一个token，然后循环，直到遇到EOS（结束标记）
- **直观体验**：大模型回答问题时一个字一个字"打"出来，就是自回归在逐个生成token
- 与之对应的是**掩码语言模型**（如BERT），通过上下文前后信息来预测缺失token

### 多模态——处理多种信息形式

- **模态**：信息的不同表现形式（文本、图像、音频、视频）
- **多模态大模型(LMM)**：能同时处理多种模态的信息
- LLM只处理文本，基础模型 = LLM + LMM

## 3. 模型训练三步链路

```
预训练(Pre-training) → 微调(Fine-tuning/SFT) → RLHF
```


| 阶段          | 目的            | 数据需求            | 核心作用                  |
| ----------- | ------------- | --------------- | --------------------- |
| **预训练**     | 从头训练，获得通用语言能力 | 海量数据（数十TB互联网文本） | 让模型具备基座知识，能做文字补全      |
| **微调(SFT)** | 在已训练模型基础上继续训练 | 较少数据（高质量Q&A对话）  | 让模型从文字补全变成支持问答格式      |
| **RLHF**    | 基于人类反馈的强化学习   | 人类对回答的排序/评分     | 实现对齐，让模型回答更安全、更符合人类偏好 |


> **为什么叫"预"训练？** 因为它是在微调之前做的训练，预训练占整体计算资源的98%。

> **RLHF全称**：Reinforcement Learning from Human Feedback

## 4. 三种核心适配技术

将现有基础模型适配到具体任务的三种方法：


| 技术       | 原理                           | 是否改变模型权重 | 难度  | 数据需求  |
| -------- | ---------------------------- | -------- | --- | ----- |
| **提示工程** | 通过设计更好的提示词引导模型               | ❌ 不改变    | 低   | 最少    |
| **RAG**  | 检索增强生成：先从知识库检索相关信息，再喂给模型生成回答 | ❌ 不改变    | 中   | 需要知识库 |
| **微调**   | 在特定数据集上继续训练，更新模型权重           | ✅ 改变     | 高   | 较多    |


### RAG工作流程

```
用户提问 → 从知识库检索相关信息 → 将信息作为上下文喂给模型 → 模型基于上下文生成回答
```

> **使用策略**：先试提示工程，不够再加RAG，还不够再微调。从最上层开始，根据需要逐步向下。

## 5. AI工程技术栈三层结构

```
┌─────────────────────────────────────────────┐
│  应用开发层                                   │
│  AI界面 / 提示工程 / 上下文构建 / 评估         │
├─────────────────────────────────────────────┤
│  模型开发层                                   │
│  推理优化 / 数据集工程 / 建模与训练 / 评估      │
├─────────────────────────────────────────────┤
│  基础设施层                                   │
│  计算管理 / 数据管理 / 服务 / 监控             │
└─────────────────────────────────────────────┘
```

- **AI应用开发者**主要工作在应用开发层
- 如果应用层搞不定，再往模型开发层深入（如微调）
- 开发AI应用时，从最上层开始，根据需要逐步向下推进

## 6. AI工程 vs ML工程


| 维度       | ML工程         | AI工程         |
| -------- | ------------ | ------------ |
| **核心任务** | 开发/训练模型      | 利用已有模型构建应用   |
| **起点**   | 从数据开始造模型     | 从现有模型开始造产品   |
| **工作流**  | 数据 → 模型 → 产品 | 产品 → 数据 → 模型 |
| **ML知识** | 必需           | 加分项，但非必需     |
| **关注重点** | 建模与训练        | 模型适配与评估      |
| **评估难度** | 封闭式输出，较易评估   | 开放式输出，更难评估   |


> 两者是**侧重点不同的两个方向**，不是父子关系。

## 7. 基础模型的应用场景

书中将应用分为8大类：


| 类别      | 消费级示例               | 企业级示例       |
| ------- | ------------------- | ----------- |
| 编程      | GitHub Copilot、代码补全 | 代码生成、文档编写   |

| 工作流自动化  | 旅行规划                | 数据提取、潜在客户挖掘 |


> **智能体(Agent)**：能够规划并使用工具的AI，是工作流自动化的核心。

## 8. 规划AI应用

### AI在产品中的角色


| 维度             | 说明                                             |
| -------------- | ---------------------------------------------- |
| **关键性 vs 辅助性** | Face ID离不开AI（关键性），Gmail没有Smart Compose也能用（辅助性） |
| **被动式 vs 主动式** | 聊天机器人是被动式，Google地图交通提醒是主动式                     |
| **动态 vs 静态**   | Face ID随用户变化更新（动态），物体检测随软件升级更新（静态）             |


### 人在回路（Human-in-the-loop）

将人类纳入AI决策过程。微软的"爬行-行走-奔跑"框架：

1. **爬行**：必须有人类参与
2. **行走**：AI可以直接与内部员工互动
3. **奔跑**：AI直接与外部用户互动

### "最后一公里"挑战

> "从0到60很容易，但从60到100极具挑战性。"

- 做出一个酷炫的演示可能只需一个周末
- 做出真正的产品可能需要数月甚至数年
- LinkedIn团队1个月实现80%体验，又花了4个月才达到95%

### AI产品的防御性

- 低门槛是双刃剑：你能做，竞争对手也能做
- 三种竞争优势：技术、数据、分发
- **数据飞轮**：率先进入市场 → 收集用户数据 → 改进产品 → 吸引更多用户
